## Findings (all NON-BLOCKING)

1. **NB · read-back's 401-retry token refresh is not abort-bounded → widens the send-accepted-but-not-recorded window** (`gmail.ts:125` calls `tokens.get(true)`; `oauth.ts:37` fetch carries no `AbortSignal`). The metadata GET itself IS bounded (`WIRE_ID_READBACK_TIMEOUT_MS=15_000` via `controller.signal`, `gmail.ts:119-123`), but the forced token refresh on a post-send 401 is not — it relies on undici's ~300s default. Reachability is vanishingly narrow: apiSend just succeeded (200) with the same cached token seconds earlier, so a 401 on the immediately-following GET requires the token to invalidate in a sub-second gap (and `EXPIRY_SKEW_MS=60s` already refreshes early), AND the refresh must then *hang* rather than error (a token-endpoint *error* is caught → returns undefined → minted-id fallback). Harm if it occurs: recordSend never runs for that ONE send → its thread mapping is never written (all replies to it lost) and the row is later mis-graded `'failed'` though the email went out once. This does NOT open a double-send (the in-flight claim is held throughout the read-back, so a racing 180s-abort→reclaim retry hits `claimSend` → 409 → retryable → cache-hit later). Same class as the pre-existing unbounded send-fetch (07-16 NB 1b) and pathological-server false-`'failed'` (07-14 NB 3), now extended to the read-back sub-path and slightly widening the known crash-window residual (store.ts:50-54). **Flag, not fix:** pass `controller.signal` into the token fetch (or bound it) so the whole read-back is ≤15s.

2. **NB · on read-back FAILURE the "dual-record safety net" does not actually save the reply — the README/store comment overstates it** (`store.ts:12-18`, README "Wire Message-ID" note, `engine.ts:104`). When the read-back fails, `wireId` is `undefined` → `canonicalId = mintedId` → `aliasIds = []` (the `canonicalId === mintedId` branch), so ONLY the minted id lands in `threads`. But the delivered Gmail message carries the WIRE id (the rewrite is the whole premise), so a reply/DSN carries the wire id → `resolveThread(wireId)` = undefined → the reply is DROPPED. The store comment "a reply matches on whichever id ended up on the delivered message" is only true in the read-back-SUCCESS case. This is NOT a regression — pre-fix, ALL Gmail replies were lost (minted-only), so read-back-failure == pre-fix behavior, and read-back-success is a strict improvement. The code comments in `gmail.ts:22-25` are accurate ("degrades reply-matching for that one send"); the README safety-net framing is the only overstatement. Operationally guarded by the README's "re-mint any send-only token before arming gmail_api" (a 403 read-back is the main failure mode and is avoided by minting `gmail.metadata`).

3. **NB · the minted alias is effectively inert for a true wire-rewriting transport** (`engine.ts:104`, `store.ts:105`). Because Gmail ALWAYS rewrites, the minted id is never on the wire → no inbound ever carries it → the minted alias in `threads` is never the matching key; the canonical (wire) id is. It is harmless (randomUUID, unguessable, maps only to its own thread — cannot mis-match) belt-and-suspenders, not the load-bearing fix. Noted only so a future reader doesn't mistake the dual-record for the mechanism that closes the bug (recording the wire id as canonical is).

4. **NB · `threads` map growth roughly doubles for Gmail mailboxes and is still unpruned** (`store.ts:20`, `:105`). Pre-existing unbounded-growth residual (one entry per send) now adds up to one more per Gmail send. Fine for the single-daemon pilot / JSON-file store; a scale-out swap (README "Scaling beyond one instance") is the eventual home. No correctness impact.

## Attacks that FAILED (why the PASS is meaningful)

- **Double-send via read-back latency (angle 1, lens 2/7).** The read-back sits INSIDE `dispatchSend`, INSIDE the `try` after `claimSend`, before `recordSend`/`releaseSend` (`engine.ts:85-111`). The claim is held for the entire read-back. The 07-14 proof still holds: the engine (`index.ts` raw `node:http`) never cancels the handler on client disconnect, so a Worker 180s abort → tick reclaim/retry racing the still-live send hits `claimSend` → `SendInProgressError` (409) → retryable → its later retry hits the populated cache → exactly one wire send. The read-back is additionally BOUNDED (15s) so it does not even approach the 300s reclaim TTL in the common path. HELD.
- **Retry re-mint / re-read-back (angle 2).** `send()`'s `if (cached) return {...}` (`engine.ts:72-73`) runs BEFORE `claimSend`, so a retry of a recorded key returns the stored canonical id with no second send and no second read-back. Read-back-failed retry returns the stored minted id (consistent with what was recorded). No path re-runs the read-back against a stale claim. HELD.
- **Dual-record persistence / ordering / collision / crash-in-gap (angle 3).** `recordSend` sets `sends[key]`, `threads[messageId]`, and every `threads[alias]` synchronously BEFORE a single `flush()` (`store.ts:103-106`) — canonical and alias are one atomic write-temp+rename, so there is no on-disk state with only one of them. A crash before the flush loses BOTH together (the known R3 residual, ledgered) → symmetric, no half-matchable state. Reload restores both (`store.test.ts` reload assertion, re-run green). Collisions: minted = `randomUUID@domain` (~2⁻¹²² collision), wire = Gmail-assigned `@mail.gmail.com` — disjoint namespaces, negligible cross-send collision. HELD.
- **Empty-string canonical trap.** `wireId ?? mintedId` would wrongly keep a `""` wire id (empty string is not nullish), but both producers guarantee non-empty-or-undefined: `parseGmailMessageId` returns undefined for an empty/absent `id`, and `fetchWireMessageId` returns `value || undefined` (empty/whitespace header → undefined, `gmail.ts:129-130`). So `wireId ∈ {non-empty, undefined}`. HELD.
- **Cross-tenant mis-thread via spoofed In-Reply-To (angle 4).** `resolveOriginal` (`classify.ts:121-142`) resolves an inbound's In-Reply-To/References/echoed-Message-IDs against `resolveThread`. The dual-record adds the minted id as a key, but (a) minted ids are randomUUID/unguessable and wire ids are Gmail-random/unguessable — an attacker can only spoof an id they legitimately received; (b) `threadId` is opaque and the platform re-scopes every event via `lookupThreadRef(ctx, ev.threadId)` under `ctx.tenantId` (`reply-processor.ts:317`), dropping any thread not in the tenant. The minted-id surface was ALREADY matchable pre-fix (it was the only stored key); the new wire-id surface is only known to the true recipient. No NEW cross-tenant path; scoping unchanged by this diff. HELD.
- **Platform deliverability-join double-count / orphan (angle 5, lens 1).** The join is `ss.message_id = e.message_id` (`deliverability.ts:257`). The dual-record is ENGINE-INTERNAL — the platform stores only the ONE canonical id in `scheduled_sends.message_id` (`tick.ts:455`, `result.messageId`); the minted alias never reaches the platform, so it can never be a wrong join key. Read-back-SUCCESS: `ss.message_id = wireId`; a bounce/complaint DSN echoes the delivered message's (wire) headers → `originalMessageId = wireId` → join matches, one count (the minted id is never on the wire so it can't appear as a DSN candidate). Read-back-FAILED: `ss.message_id = minted`, DSN carries wire, `resolveThread(wire)` = undefined → the event is dropped at the engine (never emitted) → nothing double-counted or orphaned on the platform, same as pre-fix. HELD.
- **Test vacuity / RED genuineness (angle 6, lens 5, standing lesson d342cd0).** The new `engine.test.ts` block drives the real over-time flow: `send()` a gmail_api message via a fake sender that returns the WIRE id, then a SEPARATE `poll()` whose `FakeImap` delivers a real RFC822 reply (`replyFrom` builds a genuine `In-Reply-To: <wireId>` message, `engine.test.ts:242-250`) through the real `classifyMessage` → `resolveOriginal` → `resolveThread` path with a disk-backed `EngineStore` — not a mock asserting back at itself. **Revert-fail proven:** sandbox at `scratchpad/rf-engine` (engine copy, worktree node_modules symlinked); with the fix present, 27/27 pass; with `engine.ts` + `store.ts` regenerated from HEAD (base: 4-arg `recordSend`, void `dispatchSend`), the two new tests FAIL — engine records the minted `<fccc5225…@coldstart.test>` where the test expects the wire `<CAMc35…@mail.gmail.com>` (`engine.test.ts:455`), and `resolveThread(minted)` is undefined because base `recordSend` ignores the 5th arg (`store.test.ts:46`). Non-vacuous; the fix is what turns them green.
- **apiSend return-type change (lens 7 regression ring).** `apiSend` now returns `Promise<string>`; both callers are accounted for — `gmail.ts:76` consumes it, `graph.ts:86` `await`s and discards it (safe). No third caller. Graph's `send` still returns `Promise<void>` and `dispatchSend` returns `undefined` for it (`engine.ts:203-204`), so Graph's canonical stays the minted id, correctly. HELD.
- **Timer/abort leak (angle 1).** `fetchWireMessageId` uses a fresh `AbortController` per call and `clearTimeout(timer)` in a `finally` that runs on every path (200 return, non-200 return, catch, 401-retry) — no timer kept alive, no controller reused across calls, abort-after-completion is a no-op on a settled fetch. HELD.
