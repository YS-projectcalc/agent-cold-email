## VERDICT: **SHIP-after-fixes**

The core approach — replace the manual `ENGINE_TENANTS` allowlist with a product-state `activated(tenant)` gate and a push-to-droplet credential path — is **sound and well-argued**. The two panel-03 fail-opens the gate now depends on (webhook signature, sticky-disputed) are **verified closed in current code**. But **two BLOCKING holes** must be folded into the build scope (or the design revised) before this drives a build, because both let a tenant reach real vendor spend without payment — the exact class this design exists to gate. Neither survived my attempt to refute it.

## Findings (most severe first)

### F1 — BLOCKING · lens 1/6 (fail-open) · unauthenticated `GET /checkout/simulate` is a free real-spend activation bypass under I1
**Failure scenario:** `GET /checkout/simulate?tenant=X&session=Y` is UNAUTHENTICATED (`routes/checkout.ts:24-34`, mounted on the un-authed `app` at `index.ts:44`; `/checkout` in `AUTHED_PATH_PATTERNS` does not cover the sub-path). It calls `completeSimulatedCheckout` (`engine/billing.ts:78-119`), which writes `plan = <paid tier>, billing_state = 'active'` for any pending `checkout_sessions` row — **with no gate on `STRIPE_SECRET_KEY` presence**. `CheckoutInput.plan` is paid-only (`launch|growth|scale`, `intents.ts:64-66`), so any pending session carries a paid plan. Today this is inert (real sending is structurally impossible: `factory.ts:112-116` hard-`false`). **Under I1, `activated(tenant)` = `paid && billing_state==='active' && !frozen && screening clear` → TRUE → the real EmailPort + real InboxKit provisioning.** Reachable because: (a) pre-arming test-mode tenants (the 07-12 live-verified deferred-paywall flow routinely creates pending `launch` sessions — see `ACTIVATION.md` Gate 0 junk tenant `ten_d1707c7c…`) persist in the SAME prod DO namespace the pilot arms; their `tenant`+`session` ids are in logs/records/the ACTIVATION doc itself; (b) the arming order is unconstrained — if `ENGINE_BASE_URL`/InboxKit are armed BEFORE live Stripe keys, `startCheckout` still takes the simulated branch (`billing.ts:26-64`) and the global-armed gate is already true, opening a live window. §2.5 **explicitly dismisses** this ("simulated checkout is out of the pilot flow") without closing it — a builder reading that ships the hole.
**Fix (must be in I1 scope):** fail-closed on the simulate write path when `STRIPE_SECRET_KEY` is set (guard the route AND `completeSimulatedCheckout`), OR purge pending simulated sessions at arming, OR require arming order = live-keys-before-infra AND still guard. This is the sibling of panel-03 finding #1 (webhook fail-open) reincarnated as spend-authorizing under I1.
**Cite:** `routes/checkout.ts:24-34`, `engine/billing.ts:78-119`, `index.ts:44` (unauthed mount), design §2.5:139.
**Verified:** traced route mount + auth patterns; read `completeSimulatedCheckout` (no key gate, only `billing_state != 'disputed'`); confirmed `CheckoutInput` enum paid-only.

### F2 — BLOCKING · lens 6 (design) · promo code has no redemption cap; "plan quota as spend ceiling" is false in aggregate
**Failure scenario:** §2.5 enables `allow_promotion_codes:true` + `payment_method_collection:"if_required"` on **every** checkout session (`stripe-client.ts:50-63` today sets neither), and says only "founder mints the coupon + a promotion code" — **no `max_redemptions` and no customer restriction specified.** A 100%-off code with `allow_promotion_codes` on all sessions means **any open signup** (`signup.ts` is self-serve, rate-limited to 5/min/IP but 5000 tenants/day globally, all minted `demo`) that obtains/guesses the short human-readable code completes checkout at $0 → `billing_state='active'` → activated → real spend, **OFAC-unscreened** (I5 stubbed `clear` for pilot). The design's Q2 recommendation ("accept plan quota (5-mbx) as the de-facto spend ceiling") is wrong on two counts: (1) the per-tenant ceiling is up to **60 mailboxes (scale tier)**, not 5, unless the coupon is plan-restricted (`pricing.ts:20-22`); (2) quota is **per-tenant** — there is no cap on the NUMBER of self-activating tenants, so aggregate real spend = (signups holding the code) × up-to-60 real Google Workspace mailboxes, with no absolute-$ ceiling anywhere. "Plan quota is an adequate de-facto spend ceiling" holds only for a single honest tenant, not an attacker who can open many.
**Fix:** the promo code MUST be `max_redemptions: 1` (or restricted to Mordy's Stripe customer) AND plan-restricted to `launch`; the design must specify this in §2.5 as a build/arming constraint, not leave it to "founder mints a code." Independently, an absolute-$ / tenant-count spend ceiling (SPEC §18's named "owner spend ceiling", which E1/§2.6 admit does not exist) should gate PUBLIC GA regardless of the OFAC ruling.
**Cite:** design §2.5, §2.6:148, Q2:179; `stripe-client.ts:50-63`; `signup.ts:16-22`; `pricing.ts:20-22`.
**Verified:** read the checkout body builder (no promo/collection today), signup rate limits, plan quotas, `isPaidPlanTier`.

### F3 — NON-BLOCKING · lens 6/7 · "freeze halts every spend path / guardrails inherited for free" is FALSE for the on-demand reply/followup send
**Failure scenario:** §2.1:93 claims `isLifecycleFrozen` "already halts every spend path" and §2.6:143 claims every guardrail "runs per-tick regardless and is inherited for free." But `replyToThread` (`engine/threads.ts:147`) calls `ctx.adapters.email.send` with **no** `isLifecycleFrozen` check (it is NOT among the freeze callers — only `tick.ts:150`, `provisioning.ts:157`, `campaigns.ts:27`, `byo-*`, `deliverability-actions.ts:344` are) and **no** warmup/daily-cap/`deliv_status='paused'` guard (those live inline in `runTick`). So deactivation AND deliverability guardrails for the reply/followup path depend ENTIRELY on the §2.2 port-swap making `adapters.email` sandbox when not activated — which means **§2.2 option-1 (re-evaluate `activated()` on each `buildAdapters()` with a FRESH SQL read of `status`/`billing_state`/`screening_status`) is REQUIRED, not merely "recommended."** Option-2's enumerated `invalidateAdapters()` sites reproduce exactly the miss-a-site flaw the design itself warns against, and a reader taking §2.1/§2.6 literally may skip the port-swap rigor believing `isLifecycleFrozen` already covers it.
**Cross-lane:** the concurrent warm-lead `schedule_followup` drain (same lane's review R1/R2, `docs/adversarial/warm-lead-thin-layer-design-2026-07-16.md`) must route through a cap/warmup/deliverability/suppression-guarded single-send primitive, NOT `replyToThread` — else an activated tenant's follow-ups bypass the warmup ramp (burn hazard) and, if `activated()` were cached, a frozen tenant's follow-ups keep sending.
**Cite:** design §2.1:93, §2.6:143; `engine/threads.ts:111-179` (no freeze/cap); `engine/tick.ts:150`.

### F4 — NON-BLOCKING · lens 2/8 · I3 credential-WRITE endpoint: replay/idempotency + overwrite policy unspecified
**Failure scenario:** §2.3 adds `POST /v1/mailboxes` under the single shared `ENGINE_AUTH_SECRET` (`auth.ts` — one secret for send/poll/now-write). The shared secret is adequate for *authenticity* (same trust boundary, Worker→engine), but the design specifies no replay/idempotency and no overwrite semantics for a surface that now writes the system's highest-value secrets (OAuth refresh tokens). A replayed OLD push could revert a rotated token; nothing states whether a push may overwrite an existing mailbox's creds (a reply-hijack / send-as vector if the Worker is ever tricked, since the engine's credentials map is keyed only by email with no per-tenant scoping). `router.ts:33-53` has no write route today.
**Fix:** specify an idempotency key + a rejection/last-writer policy on overwrite (and consider whether cred-rotation needs an explicit versioned replace vs blind upsert) before I3 build.
**Cite:** design §2.3:110-113; `apps/engine/src/auth.ts`, `apps/engine/src/router.ts:33-53`.

### F5 — NON-BLOCKING · lens 5 · I3 durability: creds in `engine-state.json` are silently discarded on a corrupt file
**Failure scenario:** §2.3:113 stores the cred map in `EngineStore` (the same `engine-state.json` rewritten on every send, `store.ts:114-123`). `loadState` (`store.ts:126-137`) **silently returns EMPTY on any parse error** ("missing/corrupt file → start empty"). Mixing high-frequency per-send writes with plaintext refresh tokens raises corruption exposure, and a boot after a partial/corrupt write would silently drop ALL dynamically-registered mailbox creds → every managed mailbox becomes `UnknownMailboxError` (`engine.ts:207-211`) with no signal, while env boot-map creds survive — a confusing partial outage. The design's own "must survive restart" durability requirement is undercut by silent-empty recovery.
**Fix:** fail-loud on cred-store load error (or persist creds separately from the volatile send/thread state), so a corrupt cred store is a visible boot failure, not silent mailbox loss.
**Cite:** design §2.3:113; `apps/engine/src/store.ts:126-137`, `engine.ts:207-211`.

### F6 — NON-BLOCKING · lens 1 · I3 partial-failure ordering leaves a billed-but-unsendable mailbox unspecified
**Failure scenario:** `provisionMailboxesForDomain` (`engine/provisioning.ts:43-98`) has no try/catch/rollback. The new cred-fetch+push step's failure mode is unspecified: a push failure after the InboxKit buy (real spend) + mailbox-row INSERT + metering leaves a billed mailbox the engine cannot send from. It is recoverable — `withRequestIdempotency` at the DO + the `provision` "already exists" idempotency (`mailbox-port.ts:65-67`) let a retry skip the re-buy and re-attempt the push — BUT only if the design places the cred-push so a retry actually reaches it and the mailbox row isn't treated as "done." The design (§2.3 steps 1-4) doesn't state where the push sits relative to the row insert or its retry/rollback.
**Fix:** specify cred-push ordering + retry so a partial provision is recoverable, not a permanently unsendable billed mailbox.
**Cite:** design §2.3:108-111; `engine/provisioning.ts:43-98`.

## Attacks that FAILED (why the SHIP-after-fixes is meaningful)

- **Webhook fail-open (panel-03 #1):** re-tested at HEAD — `routes/webhooks.ts:29-32` fails **CLOSED** (503) when `STRIPE_WEBHOOK_SECRET` unset; signature verified (`:39`) before any state mutation; tenant id extracted only from the signed event. The design's reliance on `billing_state` as the activation signal is safe on the webhook lane. HELD.
- **Sticky-disputed lift (panel-03 #2):** re-tested — `checkout.session.completed` (`billing.ts:200`), `customer.subscription.updated` (`:234`), `invoice.payment_failed` (`:267`), and simulated checkout (`:99`) all now guard `billing_state NOT IN ('disputed','canceling','canceled')` / `!= 'disputed'`. A renewal cannot re-activate a disputed/canceled tenant → a stopped-paying tenant stays frozen → `activated()` stays false. HELD.
- **Adapter-cache staleness (freeze direction):** the webhook is routed to the SAME DO (`idFromName(tenantId)`) that runs the tick, and the tick reads `billing_state` FRESH from SQL (`tick.ts:145-150`, `readLifecycleState`) — no cross-DO staleness. §2.2 re-evaluation closes the real-vs-sandbox cache staleness. HELD (conditional on option-1 impl — see F3).
- **Tenant-id spoof into `activated()`:** `this.tenantId` is the DO's own verified identity (constructor from the persisted row / server-minted `newId("ten")`, `tenant-do.ts:282-286`), never caller-supplied; `activated()` reads server-authoritative columns → not spoofable by client input. HELD.
- **InboxKit gate (b) exact-email assert:** converts the destructive "cancel `mailboxes[0]`" misfire (`mailbox-port.ts:123-132`) into a fail-safe "can't resolve → error, don't cancel." Closes the destructive direction (with case-normalization + gate (e) arming check). HELD.
- **InboxKit gate (c) double-charge:** the vendor's own `(domain,localPart)` uniqueness ("already exists" → idempotent success, `mailbox-port.ts:65-67`) is the real guard; the design's local record + structured-error match is a strict improvement, introduces no new double-charge. HELD.
