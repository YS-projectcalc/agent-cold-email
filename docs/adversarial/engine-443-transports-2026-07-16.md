## Findings (all NON-BLOCKING)

1. **NB · api-send has no fetch/AbortSignal timeout + grades success by exact status equality** (`api-send.ts:46-58`, `oauth.ts:37-46`). Two related robustness gaps on the duplicate-send surface:
   - (a) `if (res.status === spec.okStatus) return;` treats any *other* 2xx as a retryable failure. If a provider/proxy ever returns a non-canonical 2xx *after* actually sending, the backoff retry re-POSTs → duplicate. Low likelihood: documented success codes match (Gmail 200 / Graph 202) and the engine calls the providers directly (no proxy to rewrite the code). A `>=200 && <300` check would be safer.
   - (b) No `AbortSignal` on the token/send fetches → a hung provider connection relies on undici's ~300s default, which exceeds the Worker's 180s `ENGINE_REQUEST_TIMEOUT_MS` and reaches the 300s `SEND_CLAIM_TTL_MS` reclaim window, contradicting the `api-send.ts:17-18` "well under … the reclaim TTL" comment. This does **not** open a new double-send: the engine in-flight claim is held for the entire hang (the raw `node:http` server in `index.ts` never cancels the handler on client disconnect), so a reclaim/retry racing a still-live send gets a `409 SendInProgressError`. The only duplicate window is provider-delivered-but-client-timed-out — the same at-least-once residual the SMTP path already carries. **Fix (flagged, not applied):** bound each api-send/oauth fetch with `AbortSignal.timeout(...)` well under 180s.

2. **NB · Revoked/invalid refresh token is graded transient → 5-attempt burn + token-endpoint hammering per message** (`oauth.ts:48-49` maps every token-endpoint failure to 503). `invalid_grant` is permanent, but graded transient, so each due message burns `MAX_SEND_ATTEMPTS`=5 (Worker-side) and re-hits the OAuth token endpoint 5× before the row → `failed`. Bounded and ops-visible (rows go `failed`), but at scale (N due messages) that is ~5N `invalid_grant` token calls → possible OAuth-client rate-limiting. This is the builder's **deliberate, acknowledged** tradeoff (uniform all-transient taxonomy matching SMTP, per `errors.ts` "a mis-classified bug surfaces as a retry, not a permanent drop"). Acceptable for the 1-mailbox pilot; consider a permanent grade for `invalid_grant` before wide use.

3. **NB · Encoding tests would not catch a base64/base64url swap** (`gmail.test.ts:68`, `graph.test.ts:83`). Node's `base64` and `base64url` *decoders* are both lenient (empirically verified: decoding either alphabet with either mode round-trips to the same bytes), so decoding the emitted body proves neither Gmail's `base64url` nor Graph's `base64` *encode* choice. Current encode calls are correct (`gmail.ts:55` base64url, `graph.ts:79` base64) — this is a weak guard, not a live defect. A regression swapping the two would stay green. (Graph 200-vs-202 confusion, by contrast, **is** caught: the tests return 202/200 and exact-`okStatus` grading would throw on the wrong code.)

4. **NB · Graph app-only: no fail-fast that `send.user` equals the mailbox key** (`config.ts` superRefine, `graph.ts:52`). If an operator sets `user` ≠ the From identity, Graph sends as `user` and may rewrite the From header, diverging from the compliance/opt-out identity carried in the message. Operator-discipline only; README states `user` is the sending mailbox.

5. **NB · mint script secret exposure** (`mint-gmail-token.mjs:21`, `:74`). `client_secret` is passed as argv (visible via `ps`/shell history); the `refresh_token` is echoed to terminal scrollback. Standard for a one-time owner-run helper. The security-relevant controls are correctly present: `state` param (16 random bytes) validated on callback, `127.0.0.1`-only bind, loopback redirect_uri, reject on state-mismatch/missing-code.

6. **NB · HTTP-date `Retry-After` ignored** (`api-send.ts:80-82`). `Number("Wed, 21 Oct …")` = NaN → falls back to exponential backoff (capped at 5s). Providers typically send integer seconds; negative/huge values are already guarded and capped. Minor.

## Attacks that FAILED (why the PASS is meaningful)

- **Compliance leakage (lens 1, the top surface).** The single builder `message.ts` maps every header-bearing field of `SendEmailInput` (`listUnsubscribe`, `listUnsubscribePost`, `inReplyToMessageId`→In-Reply-To/References, explicit `messageId`, `text`=body-with-footer) — identical to the old inline SMTP mapping (the `smtp.ts` diff is a byte-for-byte extraction into `buildMailOptions`). Both API paths build from `buildRawMessage` → the same `MailComposer.compile()` nodemailer uses internally for SMTP, so parity is by construction (no second serializer). Verified by decoding the raw bytes in the gmail/graph tests: List-Unsubscribe(-Post), Message-ID, and the CAN-SPAM footer body all survive on both wires. No code path emits an API send that bypasses the builder.
- **Double-send via reclaim (lens 2).** The engine in-flight claim (`store` `inFlight` Set) wraps `dispatchSend` (`engine.ts:73-91`), transport-agnostic — the claim is taken with no `await` between the cache-miss and the claim, and released in a `finally`. A reclaim/retry racing a live API send hits the held claim → `409` → graded retryable → the retry lands after `recordSend` and returns the cached Message-ID → exactly one wire send. Holds for the API transports exactly as `engine-host-review-2026-07-14.md` round 4 proved for SMTP. The claim outlives the Worker's 180s fetch abort because the engine never cancels the handler on disconnect.
- **Silent wrong-wire / SMTP fallback (lens 3).** `dispatchSend` routes strictly by discriminator; a `gmail_api`/`ms_graph` mailbox never reaches `smtp.send`, and an unwired API transport throws an internal (transient) error rather than falling back. Config is fail-fast at boot (zod `discriminatedUnion` + `superRefine`), so no malformed/partial `send` block reaches dispatch — verified by `config.test` (7 accept/reject cases) and the engine `not wired` throw test.
- **Token cache poisoning across mailboxes (lens 4/8).** Caches are keyed by grant (Gmail `clientId:refreshToken`; Graph `mode:tenant:client:refreshToken`). Distinct mailboxes carry distinct refresh tokens → distinct caches. App-only intentionally shares one app-level token across mailboxes in a tenant — correct, because the client-credentials token is app-scoped (`.default`), not user-scoped, and per-user routing lives in the URL (`/users/{user}/sendMail`). No cross-mailbox token leakage.
- **Secrets in logs/errors (lens 4/8).** Only `index.ts:74` logs (port + mailbox count). No thrown message includes `clientSecret`/`refreshToken`/`access_token`/Bearer; `oauth.ts`/`api-send.ts` truncate only the untrusted provider *response* body (no secret) and include the token URL + a `gmail:/graph:<email>` label — never the form containing the secrets.
- **Clock-skew token race (lens 8).** `TokenCache` refreshes `EXPIRY_SKEW_MS`=60s before real expiry; `oauth.test` proves reuse-within-lifetime and refresh-past-expiry. No stale-token-on-the-wire window.
- **Regression ring on the SMTP path (lens 7).** `smtp.ts` now calls `buildMailOptions(input, messageId)`; the diff shows the removed inline object was field-for-field identical, so the SMTP wire is unchanged. The existing SMTP send/idempotency tests still pass.
