## Findings

### 1 — lens 6 (design) · NON-BLOCKING (this merge) / BLOCKING-BEFORE-ARM · A burned *fresh-standalone* BYO domain triggers an auto-BUY replacement of a domain the customer brought

- **Failure scenario (once real adapters are armed):** A tenant brings a domain they own via `register` with `domainRelationship=fresh_standalone`. It gets `breaker_tier='standard'` (`byo-intake.ts:139`). When its mailboxes' domain-aggregate complaint/bounce crosses the burn line, `evaluate()` routes it through the **existing** `REPLACE_DOMAIN` path (`deliverability.ts:167,178-187`) — `applyReplaceDomain` marks the customer's domain `'burning'` and **provisions a brand-new lookalike of the tenant's `brand`/`primary_domain`** (`deliverability-actions.ts:93-170`, `pickReplacementDomain` → `provisionDomainWithMailboxes` → `ctx.adapters.domain.buy`). There is **no `source='byo'` guard** anywhere in the burn decision or the replace action. So the platform buys (real registrar spend, real annual liability per `teardown_records`) a *different* domain — of the tenant's flagship identity, not the burned one — for a customer who opted into BYO precisely to use their own infra. Edge case: a pure-BYO tenant that never ran `setup_infrastructure` has `primary_domain=''`, so the "replacement" is a lookalike of an empty identity (fallback `slugify(brand)-r1.com`).
- **Root cause (design):** §20.1 treats `fresh_standalone` BYO as "the same shape as buying a lookalike." That is true for *deliverability isolation* but false for *ownership/replaceability* — burn-replace's precondition is that **we own and can re-buy the domain**, which is never true for a BYO domain regardless of relationship tier. The correct remedy for ANY `source='byo'` domain is `HARD_PAUSE_DOMAIN` (pause + alert), the same substitute already built for primary/subdomain tiers — keyed on `source`, not on `is_primary`/relationship.
- **Why NON-BLOCKING here:** `realAdaptersActivated=false` (hardcoded, `tenant-do.ts:290`) → `domain.buy` is `SandboxDomainPort` → the only effect in this build is a bogus *sandbox* domain row. No real spend, no PHI. Self-refuted my "BLOCKING" instinct on this basis.
- **Verification:** traced `register` → `breaker_tier` assignment → `evaluate` tier branch → `applyReplaceDomain`; grepped for any `source`/`byo` guard in the burn/replace path (none); confirmed sandbox-only via the factory call site.

Both fixes survive re-attack. The burn-replace hazard that gated BYO arming is genuinely closed at the decision layer with no reachable alternate path. The remaining items are pre-existing non-blocking residuals plus two new low-severity notes on FIX 2, none gating.

### FIX 1 (source-keyed burn routing) — HELD on all four attack angles
- **(a) source on legacy/provisioned rows.** `source TEXT NOT NULL DEFAULT 'provisioned'` (schema.ts + `addColumnIfMissing`), and `gatherDomainStats` reads `(d.source as DomainSource) || "provisioned"` (deliverability.ts:408) — a legacy/NULL row routes to the unchanged REPLACE path. No mis-route the other way: every BYO domain INSERT hardcodes `source='byo'` (byo-intake.ts:184), the NOT NULL constraint forbids a written NULL/'', and there is **no `UPDATE ... SET source` anywhere** (grep clean), so a BYO domain can never drift to 'provisioned'. HELD.
- **(b) other call sites routing on breakerTier.** `breaker_tier`/`breakerTier` now appears only as descriptive metadata (byo-intake `toRecord`, the `DomainStat.breakerTier` field, the MCP tool description). The ONLY burn/replace routing predicate is `d.source === "byo"` (deliverability.ts:180); the `continue; // NEVER falls through` (deliverability.ts:188) still blocks any BYO fall-through to the REPLACE branch. HELD.
- **(c) windowed-stats side effects on provisioned.** Gating flipped from `breakerTier === 'standard' ? {0,0} : query` to `source === 'byo' ? query : {0,0}` (deliverability.ts:409). A provisioned domain (`source='provisioned'`) still gets `{0,0}` with zero extra query — byte-identical perf/threshold behavior. The extra windowed query now correctly runs for BYO-standard (fresh-standalone) domains, which need it for the breaker. HELD.
- **(d) REPLACE_DOMAIN reachable via another entry point?** `REPLACE_DOMAIN` actions have a single producer — `evaluate()` (deliverability.ts:195) — and `applyReplaceDomain` is called only from `applyActions`'s `REPLACE_DOMAIN` case, itself reached only from `runDeliverabilitySweep`. `reporting.ts`/`ops-summary.ts` only COUNT the action for read-only rollups; no admin route, MCP tool, or ops sweep hand-constructs one. So a BYO domain (which `evaluate` can no longer emit REPLACE for) cannot reach `applyReplaceDomain`/`provisionDomainWithMailboxes` by any path. HELD.
- **RED evidence real:** `deliverability-hard-pause-domain.test.ts` inserts a fresh-standalone BYO (`is_primary=0, breaker_tier='standard', source='byo'`), drives complaints to 0.2% (well over the generic burn line), and asserts exactly one `HARD_PAUSE_DOMAIN`, **zero** `REPLACE_DOMAIN`, domain count unchanged (no purchase), domain→`paused_primary`, mailboxes→`paused`, both alerts dispatched. Fails on pre-fix code (that domain would have REPLACE'd). Genuine behavior assertion, not coverage theater.

### FIX 2 (≤1 active BYO-primary) — HELD, with two non-gating notes
- **Guard + tenant scoping correct.** `assertNoExistingActiveByoPrimary` (byo-intake.ts:82-96) counts `source='byo' AND is_primary=1 AND status='active' AND byo_status NOT IN ('rejected','abandoned')`, `WHERE tenant_id = ?`. Sequential double-registration is blocked; a `pending_consent`/`pending_dns`/`pending_kyc` primary (status='active', byo_status not excluded) correctly still blocks a second. HELD.
- **paused_primary exception is NOT abusable.** The only `domains.status` writers set `released`/`burning`/`paused_primary` — **nothing ever sets status back to `active`** (grep of all `UPDATE domains SET`). A hard-paused primary can never resume, so allowing a replacement primary while one is `paused_primary` yields at most one *active* + one dead primary (the intended "replace a burned primary" case), never two live ones. HELD.
- **abandoned/rejected exclusion is correct.** An `abandoned` primary (7-day DNS idle) keeps `status='active'` but never reached `byo_status='active'`, so it has no mailboxes and is unsendable — excluding it from the block is right; it can't be a second sendable primary. No `.one()` anywhere keys on `is_primary` (only the guard's own `COUNT(*)`), so even two `is_primary` rows crash nothing — both just receive the more-restrictive primary treatment. HELD.
- **NEW note (NON-BLOCKING) — TOCTOU race.** The guard runs at byo-intake.ts:128, *before* the two `await`s (`dnsScan.scan`, `reputation.check`) and the INSERT. Under real (network-I/O) adapters, two concurrent `is_primary` registrations could both pass the guard during the await window and both INSERT → two *active* primaries. Benign: the consequence is more guardrails, not fewer, no crash, no security/spend bypass — and it matches this codebase's own documented best-effort app-level pattern (`dashboard_views.is_default`, cited in the guard's own comment). Not triggerable in the current sandbox-only build (adapters are synchronous in-memory). Worth an atomic re-check (or a small blockConcurrencyWhile) at arm-time; does not gate.
- **NEW note (NON-BLOCKING) — pending_kyc soft-lockout.** A primary stuck in `pending_kyc` (abuse-flagged, and there is no admin-clear route in this build) keeps `status='active'` and is not excluded, so it permanently blocks the tenant from registering any new primary. Arguably correct-by-design (don't let a flagged tenant register around the flag), but flagged as an availability quirk for whoever builds the KYC-clear route.
