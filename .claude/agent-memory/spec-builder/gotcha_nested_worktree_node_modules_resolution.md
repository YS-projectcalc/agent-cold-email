---
name: gotcha-nested-worktree-node-modules-resolution
description: a worktree living under .claude/worktrees/ inside the main repo tree resolves npm-workspace packages (e.g. @scope/shared) from the OUTER main repo's node_modules, not the worktree's own, until `npm install` is run inside the worktree itself
metadata:
  type: feedback
---

In this harness, isolated worktrees are created at `<main-repo>/.claude/worktrees/agent-<id>/` — i.e. NESTED inside the main repo's own directory tree, not a sibling. Node's CommonJS/ESM module resolution walks UP the directory tree looking for `node_modules`, with no awareness of git worktree boundaries. If the worktree itself has no (or an incomplete) `node_modules`, resolution silently continues past the worktree root and finds the MAIN repo's `node_modules` several levels up — including `node_modules/@scope/shared`, which is an npm-workspace symlink pointing at the MAIN repo's `packages/shared`, NOT the worktree's own copy.

**Effect:** edits to a workspace package inside the isolated worktree (new exports, new zod schemas, etc.) are invisible to both `tsc` and `vitest` until this is fixed — you get `TS2305: has no exported member` errors that look like a mistake in your own new file, when the real problem is the resolved module isn't your file at all. Confirmed via `node -e "require.resolve('@scope/shared/package.json', {paths:[...]})"` printing the MAIN repo's path, not the worktree's.

**Fix:** run a plain `npm install` at the worktree root (respecting the existing `package-lock.json` for reproducibility) BEFORE any typecheck/test run. This creates `worktree/node_modules/@scope/shared` as a symlink to the worktree's OWN `packages/shared`, which Node resolves first (nearest ancestor wins) — correctly shadowing the outer main repo's copy.

**Why:** Why an incident happened: a "fresh-worktree test setup" brief note said only `.dev.vars` + D1 migrations were needed for the platform test suite (a real, separate gotcha — env vars are a different failure mode: `DataError: Imported HMAC key length (0)` from an unset `TOKEN_HASH_PEPPER`). It did NOT mention node_modules, because in a SIBLING-worktree layout this problem wouldn't exist. Confirm whether a worktree is nested or sibling before trusting a brief's "just needs .dev.vars" claim.

**How to apply:** In ANY nested-worktree setup (`ls -la` the worktree root — if it's under the main repo's own tree, e.g. `.claude/worktrees/`), before believing a `tsc`/`vitest` import-not-found error about your own new module: run `node -e "console.log(require.resolve('<pkg>/package.json', {paths:['<some-dir-in-worktree>']}))"` to see which copy actually resolves. If it points outside the worktree, `npm install` in the worktree root first. Also: after fixing, watch for STALE background test runs kicked off before the fix — their results are contaminated (resolved against the wrong/incomplete package) and must be discarded, not trusted as a baseline.

**Also confirmed:** concurrent test runs (yours + another agent's, on the SAME shared machine but different worktrees) is normal in this environment — `ps aux` showing unrelated `vitest`/`workerd` processes from other worktree paths is not a bug, just resource contention that can slow (or, once observed, apparently SIGTERM-kill under heavy load) a long-running foreground `npm test`. Prefer `run_in_background: true` explicitly for any `npm test` in this class of monorepo (setup alone can take 90-270s) rather than letting the tool's own 120s-timeout auto-backgrounding race with a manual `&`; never run overlapping `npm test` invocations writing to the same log file — output interleaves and becomes unusable, and requires a clean re-run to get an authoritative result.
