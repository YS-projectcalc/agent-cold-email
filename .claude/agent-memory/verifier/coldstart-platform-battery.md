**2026-07-14 human-journey + Stripe drive (live prod, not local battery):**

Fix: do multi-step crawls/transforms in a single Node script (`fetch` + regex) instead of a shell for-loop; use full paths (`/bin/ls`, `/usr/bin/curl`) if a bare invocation 127s.

**2026-07-16 apps/engine (real Gmail SMTP/IMAP daemon) Gate-1 self-send smoke — PASS, with a trap found:**
- Runtime contract: `GET /health` (no auth), `POST /v1/send` and `POST /v1/poll` (Bearer `ENGINE_AUTH_SECRET`) — see `apps/engine/README.md`. `dist/` at `apps/engine/dist/` is a plain `node apps/engine/dist/index.js` with `ENGINE_AUTH_SECRET`/`ENGINE_PORT`/`ENGINE_STATE_DIR`/`MAILBOX_CREDENTIALS_FILE` env; boots in ~1s, `/health` → `{"status":"ok","uptimeSec":N}`.
- **`POST /v1/poll` with `sinceCursor:0` against a mailbox with real history is UNBOUNDED and dangerous as a verification step** — `apps/engine/dist/imap.js`'s `fetchSince` does `uid: "${sinceUid+1}:*"` with `source: true`, i.e. fetches the FULL RFC5322 source of every message with UID > sinceCursor. On a real personal Gmail account (UID space already >147k here) this means pulling the entire inbox's raw source in one unbounded in-memory array before the HTTP response returns — it hung indefinitely and had to be `kill -9`'d. Never poll from cursor 0 against a live/real mailbox with history.
- Separately, even a *bounded* poll would show ZERO events for a same-account self-send by design, not bug: `classify.js`'s `classifyReply`→`resolveOriginal` only emits an event when the inbound message's `In-Reply-To`/`References`/other embedded Message-IDs resolve back to a KNOWN prior send-thread. A fresh self-send with `inReplyToMessageId: null` has nothing to resolve against, so it's correctly dropped (`// not a reply to any send we know -> ignore`). Don't use `/v1/poll` to verify a same-account self-send landed — verify via direct IMAP instead (below).
- **Correct verification for a real send:** confirmed via a standalone node script using the repo's own `node_modules/imapflow`, doing `client.search({header:{subject: ...}})` (server-side IMAP SEARCH, cheap) then `client.fetch(uid, {envelope:true, uid:true})` on just the matching UID(s) — compared the returned `envelope.messageId` byte-for-byte against the `/v1/send` response's `messageId`. This is fast and safe. Confirmed via `node_modules/imapflow/lib/commands/fetch.js:49-91` that imapflow fetches use `BODY.PEEK`/`.PEEK` by default — neither `source:true` nor `envelope:true` fetches mark a message `\Seen`, so read verification against a real personal inbox is safe not to alter its state.
- SSH credential pulls from the droplet (`ssh root@142.93.12.85 'cat /root/mailboxes.json'`, a live Gmail app-password) are blocked by the auto-mode permission classifier under "Production Reads" even when a teammate (team-lead) instructs it — requires the actual user's explicit runtime approval or a pre-authorized handoff of the already-pulled file; a teammate message alone does not satisfy it.
